# No global htmx error boundary: an uncaught exception 500s and blanks its swap target

Status: resolved
Category: bug
Severity: medium
Area: Transaction register / web shell

There is no application-wide error handling for htmx requests. Each htmx endpoint must hand-catch
its own exceptions; anything it misses propagates as an HTTP 500 whose (error-page) body htmx swaps
into the request's target element — blanking whatever it pointed at (typically the register rows or
the entry dock) and leaving the UI wedged until a full page refresh.

This has already bitten us twice on the register, each fixed by adding a per-endpoint `try/catch`
(commit `1b4d3bd`: phantom all-zero-base splits + wrong-currency transfer targets; see issue 05).
That is whack-a-mole: every new endpoint re-opens the same hole. This issue tracks the structural
fix — the plan's error-handling item scheduled "between stages 8 and 9"
(`implementation-plan.md`).

## Agent Brief

**Category:** bug
**Summary:** Add a global htmx-aware error boundary so an unhandled server exception surfaces as a
readable inline message instead of a 500 that blanks the htmx swap target.

**Current behavior:**
An unhandled exception thrown while servicing an htmx request (`HX-Request: true`) returns an HTTP
500 whose error-page HTML htmx swaps into the triggering element's target, erasing the register or
dock. The only recovery is a manual page refresh. Individual endpoints avoid this only where they
explicitly catch their own exceptions and re-render (e.g. the split-commit endpoint catches
`IllegalArgumentException | UnbalancedTransactionException` and re-renders the panel out-of-band).

**Desired behavior:**
An uncaught exception during an htmx request produces a controlled, htmx-friendly response that
shows the user a non-fatal error message **without** destroying the surrounding UI. Concretely: the
register/dock the user was looking at must still be on screen and usable after the error (no blank
swap, no wedged dock), and the message must be visible. A non-htmx (full-page) request should keep
whatever whole-page error behavior already exists — the boundary keys off the htmx request marker.
Genuinely expected, user-facing validation failures should continue to be handled where they occur
(inline in the panel/dock, as today); this boundary is the safety net for the *unexpected* ones,
not a replacement for local validation.

**Key interfaces:**
- A single Spring `@ControllerAdvice` / `@ExceptionHandler` surface (living in the `web` module — the
  UI shell — so every feature module's controllers are covered without a module cycle) that detects
  the htmx request (the `HX-Request` header, or an htmx response mechanism such as an OOB error
  fragment / `HX-Retarget` + `HX-Reswap` so the error lands in a dedicated error slot rather than the
  original target).
- A shared, reusable error fragment/slot in the base layout that the boundary targets, so the message
  has a stable home that is not the register or dock body.
- Existing per-endpoint validation catches stay as-is; this is added underneath them, not in place of
  them.

**Acceptance criteria:**
- [ ] An htmx POST that throws an unexpected (non-validation) exception returns a response that does
      **not** replace the register rows (`#register-rows`) or the dock body with an error page — the
      prior content is still present.
- [ ] The user sees an error message after such a failure.
- [ ] After the error, the register/dock is still interactive (a subsequent valid action works
      without a page refresh).
- [ ] A non-htmx request that throws is unaffected by the new boundary (its existing behavior holds).
- [ ] Existing inline validation flows (missing category, wrong-currency transfer target, phantom
      all-zero-base split) still render their messages inline exactly as they do now.
- [ ] Covered by a MockMvc acceptance test that drives an endpoint into an unexpected exception over
      htmx and asserts the target is preserved (a throwaway/test-only failing endpoint is acceptable
      if no production path reliably throws an *unexpected* error).

**Out of scope:**
- Per-field / on-select validation (validating a wrong-currency target the moment the account is
  picked, rather than on Save) — that is issue 05's carved-out "problem 1", a separate enhancement.
- Reworking any existing inline validation handling — leave the current per-endpoint catches in place.
- Global error handling for the MCP surface or any non-web transport.

## Resolution (2026-07-21)

Implemented in commit `e5584a3`: `GlobalHtmxErrorAdvice` in the `web` shell + `fragments/app-error
.html` + the `#app-error` slot in `layout.html` + oxblood toast styling. For an htmx request, an
unexpected exception escaping any controller is logged and returned as a 200 retargeted
(`HX-Retarget: #app-error`, `HX-Reswap: innerHTML`) to the shell error slot — never a swap into the
triggering element; non-htmx requests propagate to the default whole-page handling. Covered by
`GlobalHtmxErrorAdviceIntegrationTest`; full `./gradlew check` green; reviewed via `/code-review`.

Note: this is the safety net for *uncaught* exceptions. It is **not** what fixed issue 05 — that was
a locally-caught validation error whose OOB-only response blanked `#register-body`, fixed separately
by `HX-Reswap: none` (commit `7a9afeb`).
