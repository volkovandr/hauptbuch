# Register direction arrows sit on inconsistent sides between transfers and debt legs

Status: ready-for-human
Category: enhancement
Severity: low
Area: Transaction register — row display (§2.6), account & category columns

Transfers and person-debt legs both carry a direction arrow, but they encode the direction
**differently**: transfers keep the arrow on a fixed side and vary the glyph (`←`/`→`); debt legs
keep the glyph fixed (`→`) and vary the *side*. Side-by-side in the register the two read
inconsistently.

> Always show the arrow on the **right** of the account/person name in the **Account** column, and on
> the **left** of the name in the **Category** column. Let the glyph (`←`/`→`) carry the direction.

## Current behaviour (`register.html`)

**Account column** — person debt leg only (a real account is a bare name, no arrow):

| Direction | Renders | Arrow side |
|---|---|---|
| income / debit (they owe you more) | `→ Max` | left |
| credit (you owe them more) | `Max →` | right |

Side toggles with direction; glyph is always `→`.

**Category column** chips:

| Chip kind | Renders | Arrow side |
|---|---|---|
| transfer inbound (`From ←`) | `← Cash` | left |
| transfer outbound (`To →`) | `→ Cash` | left |
| person debt, outbound | `→ Max` | left |
| person debt, inbound | `Max →` | right |

Transfers are consistent (arrow left, glyph varies). Person legs toggle the side and pin the glyph —
that is the mismatch.

## Desired behaviour

Fix the **side per column**, and let the **glyph** carry direction (as transfers already do):

- **Account column:** arrow always to the **right** of the name — `Max →` (credit/outbound) and
  `Max ←` (debit/inbound).
- **Category column:** arrow always to the **left** of the label — `→ Max` / `← Max`, matching how
  transfer chips already sit.

Result: within a column the arrow is always in the same place, and a transfer chip and a person chip
point the same way for the same real flow. The direction each glyph should point still follows the
existing income/inbound flags — this is a rendering-position change, so confirm the glyph mapping
against register §2.6 rather than only mirroring today's `→`.

## Scope

- Template-only change in `register.html` (the two `th:if` arrow spans in the Account cell, and the
  person-chip arrow spans in the Category cell). No view-model change expected — `r.income()`,
  `chip.inbound()` etc. already carry the direction.
- Update the `register.html` comments (they document the current "→ Name on a debit, Name → on a
  credit" convention) and register §2.6 to the new fixed-side rule.
- Purely visual — add/adjust a Playwright assertion only if one already covers arrow rendering;
  otherwise no test tier applies.

## Triage outcome: ready-for-human (enhancement)

**Category corrected** from bug → enhancement: the register renders person legs and transfers with
different arrow encodings, but that difference is **deliberate**, not a defect — so this is a request
to *change* a display convention, not fix a break.

**Verified against the code and the spec.** The claim holds: person legs pin the glyph (`→`) and vary
the arrow's *side*, while transfers pin the side and vary the glyph (`←`/`→`). But the current
template already conforms to `docs/ui-transaction-register.md` §2.6, which makes the person-leg
arrow's **side** its direction channel on purpose:

- `→ Max` = debit (Max owes you more); `Max →` = credit (you owe Max more). The glyph is always `→`.
- `←` is reserved for real-account transfers and is **never** used for a person leg.
- The v0.2 rework (2026-06-22) deliberately made `Max →` mean the same thing **in either column**, so
  column and direction are decided by two independent questions.

The ticket's original "desired behaviour" (fix the side per column; let `←`/`→` carry direction) would
**break** §2.6: it introduces `← Max` and makes the arrow side column-dependent, destroying the
"`Max →` reads the same in either column" invariant that was the entire point of that rework. So the
proposed fix cannot be taken at face value.

**Why human, not agent:** the real work is a **design decision to revise §2.6 first** — settling the
person-leg direction convention (position-encoded vs glyph-encoded, and whether `←` should ever apply
to people) — and only then updating the template to match. That is authoritative-doc judgement, not a
delegatable template tweak.

**Plan (maintainer): spec first, then code.**

1. Revise `docs/ui-transaction-register.md` §2.6 to the chosen convention (bump the doc's version /
   changelog, since §2.6 has a tracked history).
2. Update the register template's person-leg arrow spans (Account cell + Category chip) to match the
   revised §2.6, keeping transfer chips consistent with it.
3. Update the `register.html` comments that document today's "→ Name on a debit, Name → on a credit"
   convention.

**Acceptance criteria:**
- [ ] §2.6 states a single, explicit rule for arrow side/glyph covering **both** person legs and
      transfers, with its example table updated to match.
- [ ] The register renders person legs and transfers per the revised §2.6 in both the Account and
      Category columns.
- [ ] The register template comments and §2.6 no longer describe the superseded convention.
- [ ] If any Playwright smoke test already asserts arrow rendering, it is updated; otherwise no new
      test tier is added (purely visual).

**Out of scope:**
- Any view-model change — `r.income()`, `chip.inbound()`, `chip.person()` already carry the direction
  and stay as-is.
- The person-debt **column-selection** rule (which leg is primary / which column it occupies) — that
  half of §2.6 is not in question here.

## Update (2026-07-21): spec revised — step 1 done

> *This was generated by AI during triage.*

Step 1 of the plan (spec-first) has landed: `docs/ui-transaction-register.md` §2.6 is reworked to
**v0.5**. The owner ratified a single, better-motivated convention (superseding the ticket's original
sketch): an arrow always sits on its label's **inner edge facing the Payee** — right of **Account**,
left of **Category** — and the **glyph** (`←`/`→`) alone carries the flow direction, so the row reads
as one continuous **Account → Payee → Category** flow line (Payee = facilitator). It is **purely
visual**; debt direction stays the sign of the running balance. Applies **uniformly** to transfer
chips and person legs; a plain expense/income category stays a **bare name, no arrow**.

**This changes what the code work is** — implement the **v0.5 §2.6** convention, not this ticket's
original "desired behaviour" prose above (which is now superseded). The v0.5 changelog entry and the
revised §2.6 table are authoritative.

**Remaining (still `ready-for-human`):** steps 2–3 — update the register template's arrow spans
(Account cell + Category chips) and the `register.html` comments to match v0.5 §2.6.
